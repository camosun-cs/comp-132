Learning Outcomes and Introduction

In this lab assignment you will learn about:

  • JavaDoc comments
  • Creating and using methods
  • Passing parameters and receiving return values
  • Splitting code into small reusable components
  • Overloading methods

Task 1: Restructure Find My Team

Transform your "Find My Team" code from Lab 3, Task 3 to use a method.

Your main method should prompt the user for a team number and print out the result league to the screen. Move the rest of the code into a new findTeam method and use parameters and return values to communicate with the main method.

Make sure to include JavaDoc comments for your new method!

Task 2: Change Machine

  1. Transform your code for calculating change from Lab 2 (Task 2: Math with Money) to use methods.

    Your main method should handle all user input and output to the screen. The actual calculation of change should be performed by a new method called makeChange which takes in an amount as a double and returns a description of the resulting change as a String.

    Starter code

    If you don't have functional code from Lab 2 to start with, you can use this version instead:

    Download starter code
  2. Add a JavaDoc comment for your makeChange method. The comment should minimally include descriptions of the method's purpose, its input parameter, and it's return value.
  3. Add static variables to keep track of the total number of each kind of coin your change machine has in stock. For example, if your change machine returns dimes and pennies, add a static int dimes and a static int pennies to your code.
  4. Add a restock method that sets the static coin counts to reasonable starting amounts, and use it from your main method to initialize them. The restock method should not need any input parameters or a return value.
  5. Add JavaDoc comments for each of your static variables and the restock method.
  6. Modify your makeChange code to decrement the coin stock counts by however many coins it uses. For example, if the method returns "2 dimes, and 3 pennies", it should decrement your dimes variable by 2, and your pennies variable by 3. If there are no longer enough coins available in the stock to make change, makeChange should return an error message instead of the change amounts. For example, if you run out of dimes, the method might return "Not enough dimes to make change."
  7. Update the JavaDoc comments for your change method to make note of the changed behaviour.

Task 3: Restructure "Play The Lottery"

  1. Recreate your "Play The Lottery" program from Lab 3 Task 2 using methods.

    Your program should be composed of at least 4 different methods:

    • a method that generates a lottery number
    • a method that prompts the user for a guess and ensures that their input is valid
    • a method that compares a guess to a lottery number and determines whether or not the guess was correct (or partially correct)
    • the main method which controls the flow of the program and determines how much money should be awarded
    Hint: use some static final constant numbers as return values from your guess-checking method to signal the different kinds of win or loss conditions.
  2. Write appropriate JavaDoc comments for all of your methods and static variables.
  3. Modify your main method to allow the player to make multiple guesses against the same lottery number. The player should be given a greater reward for matching the lottery number in fewer guesses. Try to limit your changes to the main method; if you can't, look for ways to refactor your other methods so you could use the same code for both versions of the game!
  4. Bonus: overload your lottery number generator and guess checker methods with new versions that can handle lottery numbers with a configurable number of digits. Rewrite your existing methods as special cases using these more configurable versions.
