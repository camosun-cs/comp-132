Learning Outcomes and Introduction

In this lab assignment you will practice:

  • encapsulating related data and logic into objects using classes
  • writing constructors to simplify setup and set final variables
  • using the this keyword
  • overloading constructors and methods
  • overriding the toString() method
  • creating classes based on a UML description
  • testing using a main-method class

In each of the following tasks you will be given a UML summary along with some documentation of the purpose and requirements of each method. To complete each task you must:

  1. Write a Java class that models the given description.

    Translating UML to Java

    The first few lines of a UML diagram represent member fields (variables) of the object, and tell you that field's name and type, separated by a colon. Constants are listed using UPPERCASE naming. The + at the start of each line means that the field is public.

    Constructors are listed next. Each line shows the name of the constructor (always the same as the name of the class) and an ordered list of types expected for its formal parameters (enclosed in parenthesis).

    The remaining lines each give you the name of a method, the types and order of its parameters (listed inside parenthesis), and its return type (separated by a colon).

    Your Java code should mirror the UML quite closely! Each UML line matches a top-level definition within the class block. It can be helpful to put these parts of your class in the same order as the UML so they are even easier to compare side-by-side.

    At this stage you should not add any fields, constructors, or methods that do not appear in the UML.

  2. Document your code using JavaDoc. You can use the descriptions in the Task as a starting point.

    JavaDoc Guidelines
    • Every line of code that has the public modifier should have an associated JavaDoc comment.
    • Describe the meaning of every formal parameter and method return value.
    • Write about what and why, not about how.
    • Describe things using different words than are used in the code itself.
    • Try to avoid technical jargon.
    • Give examples.
    • List and defend your assumptions.
    • Point out inputs or edge-cases that you have explicitly chosen not to handle.
    • Warn about potential areas for misunderstanding or errors.
    • Link to resources for further reading on technical details.
    • Don't repeat yourself! If the structure of JavaDoc leads to you describing the same thing in more than one place, try describing it in a different way.

  3. Write some code to test each constructor and method using a "unit testing" approach.

    All of your tests should be added to a single main-method class named TestingObjects.

    Unit Testing Principles
    • Test every part of your code. Write at least one test for every public method and constructor, and try to cover all logical paths through your code.
    • Keep each test as short and simple as possible. Use predetermined, hard-coded values, and avoid computation and branching within the code of the tests themselves. Test only one thing at a time.
    • Test things that seem trivial. These are usually the easiest tests to write and will catch the majority of simple errors in your code!
    • Test your assumptions. Make sure your program doesn't fail if your assumptions aren't met.
    • Look for edge-cases. What are some weird scenarios that you might not have considered until now? Make sure that your program doesn't crash when given malformed or unusual inputs. For example, what would happen if an input parameter was null?
    • Test what you've already fixed. If you found a bug in your code, write a test that makes sure that you never have that same bug again.
    • Make your tests automated, repeatable, and fast. Don't wait for user input, or depend on external state that may change outside the control of your testing code.
    • Make failures obvious. If something isn't working properly, make sure that it is clear what went wrong when your tests are run.
    • Run tests often. Every time you make a change to your code, re-run your tests to make sure you didn't break anything!
    Example test output

    The actual format of your output is up to you, but something like the following would be great! 

    Test: 2-argument constructor sets X and Y
Expected x == 2.0 ✅ Passed
Expected y == 3.0 ✅ Passed

Test: distance from (0,0) to (0,1) using a Point argument
Expected to return 1.0 ❌ Failed! (was 3.0)

    Since we're not using a unit testing framework, you may find yourself writing a lot of redundant code in order to display "passed" and "failed" messages. You could try creating a small UnitTest class with static methods to handle this, or leave these out and just print the expected and actual values.

Task 1: class Point

The 0-argument constructor creates a Point at (0,0). The 2-argument constructor creates a point at the given coordinates.

distanceFrom computes the straight-line distance from this Point to another coordinate, given either as a pair of (x,y) values or as another Point object.

Just a little geometry…

The straight-line distance d between two points (x1, y1) and (x2, y2) is given by the equation: 

				d2
				=
				(x2 - x1)2
				+
				(y2 - y1)2

Use Math.pow(double, double) to compute squares and Math.sqrt(double) to compute square-roots in Java.

toString should return a String that includes the class name and the X and Y coordinates; for example: "Point: [2.0, 3.5]".

Task 2: class Person

The 2-argument constructor initializes this Person's first and last name with the given values, and concatenates them to create an ID. The 3-argument constructor allows setting a custom ID value.

sayHelloTo should print a message to System.out as if this Person were greeting another person (given as the parameter). For example, if invoked with the argument "Ben" on a Person with the name "Jennifer," it should print something like: "Hello Ben, my name is Jennifer!"

Hint: avoid repeating code by calling sayHelloTo(String) from sayHelloTo(Person)

The toString method should return a value that includes the class name and the value of all four fields; for example: "Person: John Smith (C9999999), age 12".

Task 3: class Car

The constructor initializes the make, model, and year of the Car using the given values, sets the odometer reading to 0, and leaves the owner null.

The drive method prints car noises to System.out, increases the odometer reading by the given amount, and returns the ending odometer reading.

toString should return a String value that includes the class name plus the make, model, and year; for example: "Car: 2016 Tesla Model X".

Task 4: class AcademicRecord

The constructor sets the student field to the given Person, initializes the major to "undeclared", year to 1, and grades to an array with 10 elements set to zero.

calculateAverage computes the average of the values in the grades array, rounded to the nearest whole percentage.

Hint: you can use Math.round(double) to do rounding.

calculateLetterGrade should return a letter grade according the the standard Camosun grading table
